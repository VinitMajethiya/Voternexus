## 0. Agent Prime Directive

You are building **VoterNexus**, a 100% free, open-source civic-tech web platform for Indian voters. Your mandate is to build it with the same engineering rigour, accessibility discipline, and political neutrality that would be expected of a team shipping to 100 million users.

**Three rules that override everything else:**

1. **Never introduce partisan bias.** No political party names, colors, or symbols anywhere in generated UI, copy, or data. Candidate data is always treated as raw facts, never editorially framed.
2. **Never store PII.** No names, voter IDs, addresses, or location coordinates may be persisted to any database or log file. Session-only state.
3. **Never ship inaccessible UI.** Every interactive component must be keyboard-navigable and screen-reader-compatible before it is considered done.

If any task instruction contradicts these three rules, refuse the subtask and surface the conflict immediately.

## 1. Project Mental Model

Before writing a single line of code, internalize this mental model:

### 1.1 What VoterNexus Is

VoterNexus is a **phase-aware living UI**. The application detects the current election phase from a managed JSON dataset and re-skins itself — showing different primary tools, color accents, and CTAs depending on where India is in the election calendar.

```
Phase Detection → UI Theme → Feature Prominence
─────────────────────────────────────────────────
Pre-Election  (>60 days) → Blue    → Readiness tools front
Campaign      (14–60d)   → Amber   → Affidavit Decoder + Misinfo Shield
Election Week (0–7d)     → Orange  → Booth Locator + EVM Sandbox
Election Day  (today)    → Green   → Live Queue + I Voted Badge
Post-Election            → Grey    → Archive + results links
```

This phase logic lives in `lib/election-phase.ts`. All page layouts and component prominence decisions read from it. This is not a cosmetic concern — it is a core product mechanic.

### 1.2 What VoterNexus Is Not

- It is **not** a CMS or news site. No editorial content is generated by the platform.
- It is **not** a voter database. It redirects to ECI portals; it does not mirror voter data.
- It is **not** a native mobile app. It is a PWA. Invest in the PWA manifest and service worker, not a React Native branch.

## 5. Core Data Contracts

These TypeScript types are the source of truth. All components, API routes, and AI prompts must conform to them. They live in `types/index.ts`.

```typescript
// Election phase — derived from election-schedule.json + current date
export type ElectionPhase =
  | 'pre-election'      // > 60 days before polling day
  | 'campaign'          // 14–60 days before
  | 'election-week'     // 0–7 days before
  | 'election-day'      // polling day itself
  | 'post-election';    // after results

// Affidavit Decoder output
export interface AffidavitSummary {
  candidate_name: string;
  constituency: string;
  election_year: number;
  assets_inr: number | null;
  liabilities_inr: number | null;
  criminal_cases: CriminalCase[];
  education: string | null;
  profession: string | null;
  source_url: string;
  confidence_score: number;           // 0–100
  parsed_at: string;                  // ISO timestamp (session only, not stored)
}

export interface CriminalCase {
  ipc_section: string;
  plain_description: string;
  status: 'pending' | 'acquitted' | 'convicted' | 'unknown';
}

// Fact-check verdict
export type FactCheckVerdict =
  | 'TRUE'
  | 'PARTIALLY_TRUE'
  | 'FALSE'
  | 'UNVERIFIED'
  | 'OPINION';

export interface FactCheckResult {
  claim: string;
  verdict: FactCheckVerdict;
  explanation: string;                // 2–3 sentences, plain language
  sources: FactCheckSource[];
  confidence: number;                 // 0–100
  language_detected: string;         // BCP-47 code
}

export interface FactCheckSource {
  title: string;
  url: string;
  publisher: string;
  is_official: boolean;               // true = government/ECI source
}

// Booth location
export interface BoothLocation {
  booth_number: string;
  booth_name: string;
  address: string;
  district: string;
  state: string;
  latitude: number;
  longitude: number;
  is_accessible: boolean;

  queue_reported_at: string | null;
}

// Queue time report (submitted by user)
export interface QueueReport {
  booth_id: string;
  wait_level: 'short' | 'moderate' | 'long';

  // No user identity fields — ever
```

### 6.7 Affidavit Decoder (P2)

**API Route:** `app/api/affidavit/route.ts` — `runtime = 'nodejs'`

**Processing pipeline:**

```
POST /api/affidavit
  body: { url?: string, file?: base64string }
  ↓
1. Validate input — reject if neither url nor file provided
2. If url: fetch PDF server-side (never expose to client)
3. Run pdf-parse → extract text
4. If text length < 200 chars: fall back to Tesseract.js OCR (scanned PDF)
5. Sanitize extracted text (remove phone numbers, UID numbers per sanitize.ts)
6. Send to Claude claude-haiku-4-5 with structured extraction prompt (see prompt below)
7. Parse JSON response → validate against AffidavitSummary type
8. Calculate confidence_score
9. Return AffidavitSummary — do NOT cache or store
```

```
candidate affidavits filed with the Election Commission of India.

- Extract ONLY what is explicitly stated in the document.

- Do not make any political judgments about candidates.
- If a field is absent, return null. Never fabricate values.
- For criminal_cases, translate IPC section numbers into plain English descriptions.
```

### 6.9 Booth Locator (P3)

**API Route:** `app/api/booth/route.ts` — `runtime = 'edge'`

**Data strategy:**

ECI does not provide a clean REST API. Use this two-tier approach:

```
Tier 1 (preferred): Voter ID lookup
  → POST to ECI electoralsearch.eci.gov.in with voter ID
  → Scrape/parse booth details from response HTML
  → Return BoothLocation

Tier 2 (fallback): Location-based nearest booth
  → Use user GPS coords
  → Query static GeoJSON file (pre-generated from ECI PDF booth lists)
  → Return nearest 3 booths by Haversine distance
```

The static GeoJSON fallback (`public/data/booths.geojson`) must be generated by a maintainer script (`scripts/generate-booth-geojson.ts`) and re-run each election cycle.

**Frontend component:** `components/civic/BoothLocator/`

- Map: `react-leaflet` with OpenStreetMap tiles (no API key)
- Booth pin: custom SVG marker (ballot-box icon)
- "Get Directions" → deep-link `https://maps.google.com/?daddr=[lat],[lng]`
- Accessibility filter: checkbox group — filters markers on map client-side
- Queue status badge overlaid on pin color

**Privacy rules:**

- GPS coordinates are used client-side for Tier 2 distance calculation — never sent to the server
- Voter ID is sent to the server only for the Tier 1 lookup, not stored or logged

### 6.10 Queue Time Reporter (P3)

**API Routes:**

- `GET /api/queue?booth_id=X` — returns current queue status for a booth
- `POST /api/queue` — submits a new report `{ booth_id, wait_level }`

**Redis data model:**

```
Key: queue:{booth_id}
Value: JSON array of last 20 reports: [{ wait_level, ts }]
TTL: 2 hours (auto-expire after election day traffic drops)
```

**Aggregation logic (in route handler):**

```typescript
// Only consider reports from the last 45 minutes
const recent = reports.filter(r => Date.now() - r.ts < 45 * 60 * 1000);
if (recent.length === 0) return 'unknown';

// Mode of recent reports
const counts = { short: 0, moderate: 0, long: 0 };
recent.forEach(r => counts[r.wait_level]++);
return Object.entries(counts).sort((a, b) => b[1] - a[1])[0][0];
```

**Anti-abuse:**

- IP rate limit: 1 POST per booth_id per 15 minutes per IP (Upstash sliding window)
- Outlier rejection: if new report deviates >2 levels from current consensus, flag it and do not count it

## 7. API Route Standards

Every API route must follow this pattern without exception:

```typescript
// app/api/example/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { z } from 'zod';
import { rateLimit } from '@/lib/rate-limit';

export const runtime = 'edge'; // or 'nodejs' if needed

const RequestSchema = z.object({
  // Define with Zod. Always.
});

export async function POST(req: NextRequest) {
  // 1. Rate limiting — always first
  const rateLimitResult = await rateLimit(req, { requests: 10, window: '1h' });
  if (!rateLimitResult.success) {
    return NextResponse.json({ error: 'Rate limit exceeded' }, { status: 429 });
  }

  // 2. Parse and validate body — always Zod
  const body = await req.json().catch(() => null);
  const parsed = RequestSchema.safeParse(body);
  if (!parsed.success) {
    return NextResponse.json({ error: 'Invalid request', details: parsed.error.flatten() }, { status: 400 });
  }

  // 3. Business logic
  try {
    const result = await doWork(parsed.data);
    return NextResponse.json(result);
  } catch (err) {
    console.error('[api/example]', err);
    return NextResponse.json({ error: 'Internal server error' }, { status: 500 });
  }
}
```

**No route may:**

- Log request body contents (could contain voter IDs or personal data)
- Return stack traces to the client
- Skip input validation
- Call the Anthropic API without a timeout (set `AbortSignal.timeout(15000)`)

## 8. AI Integration Standards

All AI calls go through `lib/ai-client.ts`. Never call the Anthropic SDK directly from a component or route handler.

```typescript
// lib/ai-client.ts
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

export async function runStructuredExtraction<T>(
  systemPrompt: string,
  userContent: string,
  schema: ZodSchema<T>,
  options?: { maxTokens?: number; timeoutMs?: number }
): Promise<{ data: T; confidence: number }> {
  const controller = new AbortController();
  const timeout = setTimeout(
    () => controller.abort(),
    options?.timeoutMs ?? 15000
  );

  try {
    const response = await client.messages.create({
      model: 'claude-haiku-4-5-20251001',
      max_tokens: options?.maxTokens ?? 1000,
      system: systemPrompt,
      messages: [{ role: 'user', content: userContent }],
    }, { signal: controller.signal });

    const text = response.content[0].type === 'text' ? response.content[0].text : '';
    const parsed = JSON.parse(text.replace(/```json|```/g, '').trim());
    const validated = schema.parse(parsed);

    return { data: validated, confidence: calculateConfidence(response) };
  } finally {
    clearTimeout(timeout);
  }
}
```

**Model selection:** Always use `claude-haiku-4-5-20251001` for speed and cost. Do not use Sonnet or Opus — the structured extraction tasks do not require it.

**Prompt storage:** All prompts live in `lib/prompts/`. They are TypeScript string exports, not markdown files. They are committed to the repository and open for community review.

**Confidence scoring:**

```typescript
function calculateConfidence(response: Anthropic.Message): number {
  // Use stop_reason and token usage as proxy signals
  if (response.stop_reason !== 'end_turn') return 40;
  const ratio = response.usage.output_tokens / response.usage.input_tokens;
  if (ratio < 0.05) return 55; // Very short output — likely incomplete
  if (ratio > 0.5) return 70;  // Healthy output relative to input
  return 85;
}
```

### 9.2 Bento Grid Pattern

All feature cards use this pattern. Never use plain `grid-cols-3` — always `bento`:

```tsx
// components/layout/BentoGrid.tsx
export function BentoGrid({ children }: { children: React.ReactNode }) {
  return (
    <div className="grid grid-cols-1 md:grid-cols-4 gap-4 auto-rows-auto">
      {children}
    </div>
  );
}

// Span variants applied to cards:
// col-span-1          → small card
// md:col-span-2       → medium card
// md:col-span-4       → full-width card
// md:col-span-2 row-span-2 → tall card
```

### 9.3 Phase-Aware Color Injection

The `PhaseProvider` injects a CSS custom property into the document root:

```typescript
// In PhaseProvider.tsx
useEffect(() => {
  document.documentElement.style.setProperty(
    '--color-phase-accent',
    PHASE_COLORS[phase]
  );
}, [phase]);
```

All phase-aware UI elements reference `var(--color-phase-accent)` via a Tailwind arbitrary value `text-[var(--color-phase-accent)]`. This is the only approved mechanism for phase-driven color changes.

### 10.1 Unit Tests (Vitest)

Every `lib/*.ts` utility must have a corresponding `tests/unit/*.test.ts` file.

### 10.2 E2E Tests (Playwright)

Critical user journeys must have Playwright E2E coverage:

```
tests/e2e/
├── eligibility-quiz.spec.ts        # Full happy path + "not sure" path
```

Run accessibility scan in CI:

```typescript
// In accessibility.spec.ts
import AxeBuilder from '@axe-core/playwright';

test('home page passes WCAG 2.1 AA', async ({ page }) => {
  await page.goto('/');
  const results = await new AxeBuilder({ page })
    .withTags(['wcag2a', 'wcag2aa', 'wcag21aa'])
    .analyze();
  expect(results.violations).toEqual([]);
});
```

- Any E2E accessibility scan has WCAG AA violations
- TypeScript compilation has errors

## 11. Internationalization (i18n)

Use `next-intl`. All user-facing strings live in `messages/{locale}.json`. No hardcoded strings in JSX.

```tsx
// ✅ Correct
import { useTranslations } from 'next-intl';
const t = useTranslations('quiz');
<h1>{t('question_1_title')}</h1>

// ❌ Wrong — never do this
<h1>Are you 18 years or older?</h1>
```

Message key naming convention: `{feature}.{component}.{element}`

```json
// messages/en.json (abbreviated)
{
  "quiz": {
    "progress": "Question {current} of {total}",
    "q1_title": "Are you 18 years or older?",
    "q1_yes": "Yes",
    "q1_no": "No",
    "result_eligible_title": "You're all set to vote! 🎉",
    "result_eligible_body": "You're eligible. Next step: confirm your documents."
  },
  "checklist": {
    "title": "What to Bring on Polling Day",
    "ready_banner": "You have {count} document(s) ready ✓",
    "cta_lost_id": "Lost your ID? Here's what to do →"
  }
}
```

When adding a new string: add to `en.json` first, then add a placeholder (`"[TODO: translate]"`) to all other locale files. The CI will flag `[TODO: translate]` strings as warnings but not failures.

## 14. What To Do When Stuck

**If ECI APIs are unreliable or blocked:**
→ Fall back to static GeoJSON/JSON data. Document the gap in `DECISIONS.md`. Open a GitHub issue tagged `data-source`.

**If a PDF is scanned and OCR confidence is low:**
→ Return the result with `confidence_score < 60` and render the yellow warning badge. Never suppress low-confidence results — always show them with appropriate caveats.

**If an AI response does not parse as valid JSON:**
→ Retry once with an explicit `"Respond ONLY with JSON, no other text."` appended. If it fails again, return a 503 with `{ error: 'AI processing failed', retry: true }`. Never return partial or malformed data.

**If you are unsure whether a UI decision introduces political bias:**
→ Default to removing it. Bias errors are unrecoverable. Missing features are fixable.

**If a new feature is requested that requires storing user data:**
→ Refuse. Log the request in GitHub Discussions under the label `privacy-review` for the maintainer team to evaluate.
